### 2.2. Система сборки

Используется `setuptools` через декларативный `pyproject.toml`:

| Файл | Назначение |
|------|------------|
| [`requirements.txt`](requirements.txt) | список рантайм-зависимостей с пиновкой версий |
| [`pyproject.toml`](pyproject.toml) | метаданные пакета `culinary-blog`, зависимости, точка входа `culinary-blog`, доп. группа `docs` для Sphinx |

Раскладка по `src/`-layout (рекомендованная):

```
src/
└── culinary_blog/
    ├── __init__.py
    ├── main.py
    ├── ...
    └── templates/  static/   # включаются в пакет через package-data
```

Собрать пакет:

```bash
pip install build
python -m build
# появятся dist/culinary_blog-0.1.0.tar.gz и .whl
```

После установки (`pip install dist/culinary_blog-0.1.0-py3-none-any.whl`) появляется
консольная команда `culinary-blog`, которая поднимает Uvicorn.

**Пиновка версий:** все версии зафиксированы. Это даёт воспроизводимую сборку.
Пиновка `bcrypt==4.0.1` отдельно — потому что `passlib 1.7.4` несовместим с `bcrypt >= 5.0` (passlib делает внутренний self-test, который падает на новой версии). Обнаружено на этапе sanity-проверки.

### 2.3. Docstrings

Соблюдена рекомендация PEP 257 (литература №5 в части 2): docstring у модуля и у публичных
функций/методов/классов. Пример:

```python
def verify_password(password: str, password_hash: str) -> bool:
    """Проверяет, что пароль соответствует хешу."""
    return _pwd.verify(password, password_hash)
```

### 2.4. Sphinx — документация

В каталоге [`sphinx_docs/`](sphinx_docs):

- `conf.py` — конфигурация. Указано:
  - `project`, `author`, `release`, `language = "ru"`,
  - расширения: `sphinx.ext.autodoc`, `sphinx.ext.napoleon` (понимает Google/Numpy-style docstring), `sphinx.ext.viewcode` (вставляет исходник в страницы),
  - `sys.path.insert` на `src/`, чтобы autodoc увидел пакет.
- `index.rst` — заглавная страница с `toctree`.
- `modules.rst` — `automodule` для каждого модуля пакета. Sphinx сам пройдёт по docstring'ам и сгенерирует страницы.

Собрать HTML-документацию:

```bash
pip install -r requirements.txt
python -m sphinx -b html sphinx_docs sphinx_docs/_build/html
# открыть sphinx_docs/_build/html/index.html
```

Альтернатива: `make -C sphinx_docs html` (по `sphinx_docs/Makefile`).

Сборка проверена локально, проходит успешно: «`build succeeded`».

## Часть 3. Развёртывание

### 3.1. Dockerfile

Файл [`Dockerfile`](Dockerfile):

- Базовый образ `python:3.11-slim` (Linux-окружение, как требует методичка).
- Устанавливает системные `gcc` и `libpq-dev` (нужны для сборки `psycopg`).
- Ставит зависимости из `requirements.txt`, затем сам пакет (`pip install -e .`).
- Создаёт каталог под загружаемые фото (`/app/media/uploads`).
- Запускает Uvicorn на `0.0.0.0:8000`.

**Параметры подключения берутся из переменных окружения** (`DATABASE_URL`, `SECRET_KEY`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`) — это прямое требование методички для части 3.

### 3.2. docker-compose.yml

Файл [`docker-compose.yml`](docker-compose.yml) поднимает 2 сервиса:

| Сервис | Образ | Порт | Назначение |
|--------|-------|------|------------|
| `db` | `postgres:15-alpine` | 5432 | PostgreSQL, тома данных в `pgdata` |
| `app` | (build из Dockerfile) | 8000 | FastAPI |

Связь между сервисами:
- `app.depends_on.db` с `condition: service_healthy` — приложение не стартует, пока БД не пройдёт `pg_isready`.
- `DATABASE_URL` указывает на host `db` (имя сервиса в сети compose).
- Том `./media:/app/media` пробрасывает загруженные фото из контейнера на хост.

Запуск:

```bash
docker compose up --build
# http://localhost:8000
```

### 3.3. Развёртывание в облако (необязательный пункт)

Готовый Dockerfile позволяет деплоить на любую платформу, поддерживающую контейнеры.
Heroku, упомянутый в методичке, отказался от бесплатного тарифа в 2022 г.
Актуальные бесплатные альтернативы — **Render** и **Railway**.

**Render (web service + managed Postgres):**

1. Зарегистрироваться на [render.com](https://render.com).
2. New → PostgreSQL → создать managed-инстанс, скопировать `Internal Database URL` (вид `postgresql://...`).
3. Заменить схему на `postgresql+psycopg://` — это тот же URL, но для драйвера `psycopg` v3.
4. New → Web Service → подключить GitHub-репозиторий.
5. Указать **Environment = Docker** (Render возьмёт `Dockerfile`).
6. В Environment Variables прописать:
   - `DATABASE_URL` = переделанный URL из шага 3,
   - `SECRET_KEY` = длинная случайная строка,
   - `ADMIN_EMAIL`, `ADMIN_PASSWORD`.
7. Нажать Deploy. Render сам соберёт образ и поднимет сервис на публичном HTTPS-домене.

**Railway:** аналогично — `railway init`, `railway add` для PostgreSQL, переменные окружения через web-консоль или `railway variables set ...`, потом `railway up`.

Деплой возможен, потому что:
- приложение не зависит от локальных путей (всё через `Path`),
- настройки берутся из переменных окружения,
- порт читается через `--port 8000`, что соответствует ожиданиям облачных платформ.

## Архитектура и поток запроса

```
Browser
   │
   │ HTTP
   ▼
┌──────────────────────────────────────────────┐
│  FastAPI (uvicorn)                            │
│   ├── SessionMiddleware (cookie с подписью)   │
│   ├── /static  → StaticFiles  (CSS, JS)       │
│   ├── /media   → StaticFiles  (загруж. фото)  │
│   │                                            │
│   ├── routers/pages.py   GET-эндпоинты, рендер│
│   │     Jinja2 → HTML                         │
│   ├── routers/auth.py    /register, /login    │
│   ├── routers/recipes.py CRUD рецептов, like, │
│   │                       favorite             │
│   └── routers/admin.py   /admin/*             │
└─────────────────┬─────────────────────────────┘
                  │ SQLAlchemy
                  ▼
            PostgreSQL (или SQLite в dev)
```

### Слои

| Слой | Файлы | Назначение |
|------|-------|------------|
| Конфигурация | `config.py` | Чтение `.env`, пути, секреты |
| Доступ к данным | `database.py`, `models.py` | Engine, Session, ORM-модели |
| Безопасность | `auth.py` | bcrypt-хеш, `current_user`/`require_user`/`require_admin` |
| Маршруты | `routers/*.py` | Бизнес-логика и рендер |
| Представление | `templates/*.html` | Jinja2 |
| Поведение | `static/js/*.js` | Лайки/избранное через fetch, динамика формы |
| Стили | `static/css/styles.css` | Палитра по макету |

### Поток «поставить лайк»

1. Пользователь жмёт `👍` на `recipe.html`.
2. JS в `static/js/recipes.js` ловит клик, отправляет `POST /recipes/:id/like` с cookie.
3. FastAPI извлекает `user_id` из сессии в `auth.current_user`.
4. `recipes.toggle_like` проверяет: нет лайка → создаёт, есть → удаляет (toggle).
5. Возвращает JSON `{liked, count}`.
6. JS обновляет кнопку и число лайков без перезагрузки.

### Сущности БД

| Таблица | Поля (главное) | Связи |
|---------|----------------|-------|
| `users` | email, name, password_hash, is_admin, is_blocked | → recipes |
| `recipes` | title, description, image_path, author_id | ← ingredients, steps, likes, favorites |
| `ingredients` | recipe_id, name, amount, position | |
| `steps` | recipe_id, text, position | |
| `likes` | user_id, recipe_id (UNIQUE) | |
| `favorites` | user_id, recipe_id (UNIQUE) | |

Таблицы создаются автоматически в обработчике startup (`Base.metadata.create_all`).
Миграции (Alembic) сознательно не подключены — для учебного проекта избыточно.

### Аутентификация

Подписанные cookie-сессии через `starlette.middleware.sessions.SessionMiddleware`.
Это самый простой вариант для server-rendered приложения: после `/login` в cookie кладётся `user_id`, далее каждый запрос → `current_user(request)` достаёт его и подгружает модель.

## Как запустить и проверить

### Способ 1. Локально (без Docker)

```bash
cd ТРПП_итоговый_проект
python -m venv .venv
.venv\Scripts\activate            # Windows
# . .venv/bin/activate            # Linux/Mac

pip install -r requirements.txt
copy .env.example .env             # Windows
# cp .env.example .env             # Linux/Mac

# поднять PostgreSQL отдельно (или подменить на sqlite через DATABASE_URL)
python -m culinary_blog            # http://localhost:8000
```

Войти в админку: `admin@example.com` / `admin123` (из `.env`).

### Способ 2. Docker Compose

```bash
docker compose up --build
# http://localhost:8000
```

Compose сам поднимет PostgreSQL и подождёт его готовности.

### Способ 3. Установленный пакет

```bash
pip install build
python -m build
pip install dist/culinary_blog-0.1.0-py3-none-any.whl
culinary-blog                      # появилась команда из pyproject.toml
```

### Проверка функционала (smoke-тест проведён, всё работает)

| Шаг | Ожидание |
|-----|----------|
| Открыть `/` без логина | 200, главная без ошибки |
| Открыть `/favorites` без логина | 401 (не пускает) |
| `POST /register` с email/паролем | 303 → `/`, в cookie появилась сессия |
| `POST /recipes` (форма + файл) | 303 → `/recipe/<new-id>` |
| `POST /recipes/<id>/like` | `{liked: true, count: 1}` |
| `POST /recipes/<id>/favorite` | `{in_favorites: true}` |
| `GET /recipe/<id>` | 200, страница с заголовком рецепта |
| Зайти под admin, открыть `/admin` | 200, видны таблицы рецептов и пользователей |

### Сборка документации

```bash
python -m sphinx -b html sphinx_docs sphinx_docs/_build/html
start sphinx_docs\_build\html\index.html    # Windows
```
